Semantic Versioning

Modern software and API's track their version number and using Semantic
Versioning, A three number identity that specifies breaking changes,
feature additions and bug fixes.

In response to Semantic Version video

One thing I noticed while watching the video is that there was alot of emphesis put on the importance of semmantic version in regards to 'public facing API'. This makes sense when I think of software language versions or library versions but much less useful for more traditional software. A lot of the software I use on my computer have no API to interface with but they follow a similar versioning. The importance of semmantic versioning really seems to shine when considering if you are fine not upgrading a core library or framework for your project.

Newer is not better if you breaks your software. For example the python community was divided when python 3 was released. Python 3 broke backwards compatibility so the majority of python modules were incompatible from the start. No one wanted to upgrade to Python 3, even as it began to eclipse Python 2 in terms of performance and features. People are still sour about it. Same thing with the Angular JS framework in regards to AngularJS vs Angular 2/4
